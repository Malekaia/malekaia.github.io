Why type declarations for Rust constants should be unnecessary

Minor inconsistencies in an otherwise perfect language

By Malekai, 09/06/2022 4 minute read By Malekai • 4 minute read 09/06/2022
Author:

Introduction:

In Rust mutable variables are created using the let keyword. They can be defined with or without type annotations:

let var_name = 10;
let var_name: i32 = 10;

However, when dealing with const variables you must always specify a type:

const VAR_NAME: i32 = 10;

The Rust docs state that:

Constants must always be statically typed.

Rust, Constant Items

Otherwise, the compiler panics and throws the following error:

error: expected `:`, found `=`
--> src/main.rs:5:11
3 |   const VAR_NAME = 10;
                ^ expected `:`

For many, this seemingly arbitrary rule can be confusing, especially when you consider the fact that the Rust compiler capable of inferring the type of a constant / static variable.

Rust has two different types of constants which can be declared in any scope including global. Both require explicit type annotation.

Rust, Constants

Inconsistent rules:

The main argument is that Rust cannot infer types for constants, however this doesn't seem to be an issue for variables declared with let, even for Complex types.

If Rust might infer that a const variable with a value of 22 is of type i32 when it should be usize, why wouldn't it make the same mistake for let variables? Simple, inference can only occur when the code that is using that variable is used, without using a variable its type will be difficult to infer, after all the compiler can't know what you're doing if you've never done it.

For example, when you write: let x = 42, the compiler infers the type of that integer from the way in which it is subsequently used. If the code expects a variable of type u8, then the compiler will infer that x is of type u8 and if the code expects a variable of type u32, then the compiler will infer that the value is of type u32.

However for both let and const variables, if there are multiple possible types for the variable then the compiler will demand that you specify a type.

This is understandable, but why enforce strict type annotations for constants at all times when both mutable and immutable variables are able to infer types in certain scenarios?

On the flip side, it's inconsistent to enforce strict type annotations on global let variables but not on local let variables that never leave their host scope. This is understandable for public or global data, as it makes sense to enforce strict type annotations to prevent type errors from occurring in other source files, but enforcing different scope based rules for mutable and immutable variables will only lead to confusion.

Potential solutions:

Based on the arguments above, the rules should be standardised for both mutable and immutable variables:

  1. Type inference should be limited variables with scalar types
  2. Type inference should be limited to local variables
  3. Type inference should be limited to values with lifetimes that end before the program does
  4. Global variables should be explicitly typed

Closing thoughts:

Like many, I enjoy working with Rust and regularly use it. I wrote this opinion piece not to bash the language but to discuss my opinions. While I acknowledge my bias in favour of these arguments, I also understand what happens when established languages change long-established rules. [1][2]

As always, take other people's opinions with a pinch of salt and do your research before forming your own.
